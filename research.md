Open-Source Privacy Controls for Semantic Routing with vLLM
Executive assessment
Your concern is technically valid: if the sensitivity or complexity decision is itself made by a remote SaaS model, then the system has already leaked the unredacted prompt across the trust boundary before routing happens. In the Athena-era architecture, vLLM Semantic Router sits in front of model backends as Client → Envoy Proxy → ExtProc → Backend Model, and the public materials describe it as a routing layer across local, private, and frontier models under cost, privacy, latency, and safety constraints. Athena also added safety, semantic caching, memory, retrieval, and long-context handling, while its classifier stack includes PII detection and token labeling for safety-sensitive spans. That means the right privacy architecture is to keep risk scoring local, and treat any external model call as a second-stage action that occurs only after sanitization or explicit policy approval. 

The strongest answer I found is not a single turnkey product, but a composable open-source stack that already exists by 2026: a local detector/redactor such as Microsoft Presidio paired with GLiNER or GLiNER2-PII for higher-recall span detection; a gateway/guardrail layer such as LiteLLM, NeMo Guardrails, Guardrails AI, LLM Guard, or OpenGuardrails; a local memory and retrieval layer such as LangGraph, Letta, Mem0, or MemMachine backed by a local vector store such as Qdrant; and hard egress controls enforced with Envoy ext_proc, OPA-Envoy, Kubernetes NetworkPolicy, Cilium Egress Gateway, or Istio. In practice, that is enough to build a fully open-source “redact-before-SaaS” reference architecture around vLLM Semantic Router today. 

Why the leakage risk is real in a semantic-router design
Red Hat’s Athena walkthrough makes the risk concrete: the router is an OpenAI-compatible front end that can send easy traffic to a local model and hard traffic to a cloud model such as Gemini, OpenAI, or Claude, all without the client changing its API behavior. If the content-based decision that chooses the “hard traffic” lane is made remotely, the privacy boundary is already broken. The fact that Athena already exposes local signal extraction, local mmBERT-based classifiers, local PII detection, and token labeling for sensitive spans strongly argues that sensitivity classification should stay inside the isolated cluster or laptop, not be outsourced to the very providers you are trying to protect data from. 

Recent privacy papers support that design choice. PPRoute describes privacy-preserving LLM routing as an underexplored problem and presents what it calls the first systematic routing framework under MPC, precisely because third-party routing creates a chain-of-control problem for sensitive queries. PRISM routes prompts among edge-only, cloud-only, and collaborative paths after edge-side sensitivity profiling, then perturbs sensitive entities locally before any cloud participation. The practical lesson from both papers is simple: if routing itself is privacy-sensitive, the first pass over content must happen on-prem. 

There is a second, subtler risk: reasoning and observability layers can leak even when final answers look safe. Leaky Thoughts shows that reasoning traces frequently contain sensitive user data and that more test-time reasoning can enlarge the privacy attack surface. Chain-of-Sanitized-Thoughts goes further and argues that hidden reasoning cannot be treated as harmless internal state, because CoT traces are often logged, displayed in dashboards, or stored in analytics systems. For a semantic-router demo, that means prompt inspection, safety analysis, traces, semantic caches, and debug logs all need the same privacy treatment as the final outbound request. 

What recent research implies for your architecture
The best recent paper for your exact “redact before third-party inference” use case is Anonymous-by-Construction. It benchmarked local anonymization against Microsoft Presidio, Google DLP, and other baselines, and found that an on-premise anonymization layer with type-preserving substitution can prevent sensitive content from reaching third-party APIs while preserving semantic utility for downstream Q&A. The design implication is important: for semantic routing, pseudonymization or placeholder substitution is generally better than blunt deletion, because routing and answer quality both depend on preserving task structure and entity roles. 

Research on memory and retrieval points in the same direction: minimize the amount of raw context that ever needs to be sent off box. Knowledge Access Beats Model Size finds that persistent agents often receive highly repetitive user-specific queries, and that memory-augmented routing can let a lightweight local model recover much of a large model’s usefulness while sharply reducing inference cost. Sufficient Context similarly shows that knowing whether retrieved context is actually sufficient can improve selective generation and guided abstention. Together, these papers suggest that better local memory and retrieval can reduce the number of cases that truly need a frontier SaaS model at all. 

On the output side, Privacy-Aware Decoding is also relevant. It shows that even if retrieval is protected, the generation stage can still become a leakage channel, and proposes an inference-time defense that reduces leakage without changing the retriever or the underlying model. You do not need to adopt PAD itself to use the idea: in production terms, it means output scanning and response validation still matter after a sanitized prompt leaves the cluster, because a remote model can reconstruct, infer, or regurgitate private information from partial context. 

The research on agent memory also favors “local-first, ground-truth-preserving” memory systems over heavy LLM-based extraction pipelines. MemMachine stores raw conversational episodes, minimizes routine LLM extraction, and reports strong accuracy-efficiency tradeoffs with lower token usage than Mem0 in matched comparisons. That matters because every extra summarization or extraction pass is another opportunity to distort, over-share, or leak sensitive state. For your use case, memory should stay local and only the smallest policy-approved slice of context should ever be exposed to a SaaS fallback. 

Open-source tools that fit this problem
Redaction and sensitive-span detection
Microsoft Presidio is the strongest foundational open-source choice for pre-egress redaction. Its official docs describe fast identification and anonymization for text and images, predefined or custom recognizers, external PII models, Docker and Kubernetes deployment, and explicit support for pseudonymization mappings. Just as importantly, the docs warn that automated detection is not guaranteed to catch everything and should be combined with other protections, which matches the layered design this problem needs. Presidio is MIT-licensed. 

GLiNER is an excellent companion to Presidio when you need local, CPU-friendly entity detection for custom taxonomies such as employee IDs, project codenames, cluster names, internal product names, or organization-specific secrets. The project is Apache-2.0 licensed, supports zero-shot entity extraction, can run locally or as a Ray Serve deployment, and is explicitly positioned as lightweight enough for consumer hardware. The new GLiNER2-PII paper is especially promising because it introduces a multilingual PII model for 42 entity types and reports the best span-level F1 on the SPY benchmark among compared systems, including OpenAI Privacy Filter and multiple GLiNER baselines. For your scenario, Presidio + GLiNER/GLiNER2-PII is the most credible all-open-source redaction core I found. 

LLM Guard is not a substitute for a DLP engine, but it is a very useful supplement. Its OSS toolkit includes prompt-side scanners such as Anonymize, PromptInjection, Secrets, Regex, and TokenLimit, and output-side scanners such as Sensitive, Deanonymize, Regex, and Relevance. That makes it a strong second pass around the main Presidio/GLiNER pipeline, especially for secret detection, prompt-injection defense, and output validation. It is MIT-licensed. 

Guardrails and gateway layers
NeMo Guardrails is the most mature open-source framework I found for programmable enforcement around LLM calls. NVIDIA’s docs say it intercepts inputs and outputs, applies configurable checks, and supports input, retrieval, dialog, execution, and output rails. The guardrail catalog explicitly includes PII handling, jailbreak detection, topic control, and agentic security, and the framework can run as a local API server, in Docker, or as production microservices. For teams already building agent flows, its LangChain and LangGraph integrations are particularly relevant. 

Guardrails AI is also strong if your stack is Python-heavy and validator-driven. The project is Apache-2.0 licensed, supports input and output guards, can run as a standalone service, and has validators that specifically use Presidio and GLiNER for PII detection and anonymization. Its server mode is useful if you want a dedicated “validation hop” between vLLM Semantic Router and any external backend. 

LiteLLM is highly relevant because it is an open-source, self-hosted, OpenAI-compatible AI gateway, which matches the API shape you are already using. The docs show a hook system that can modify or reject incoming requests before the LLM call, plus a built-in Presidio PII guardrail that supports pre_call, post_call, and logging_only modes. LiteLLM also has a complete tutorial for Presidio-based masking of LLM calls. The main caveat is that some safety features are not part of OSS—for example, its secret-detection redaction page is explicitly marked enterprise-only—so I would treat LiteLLM as a gateway shell with open hook points, not as the sole privacy engine. 

OpenGuardrails is the closest thing I found to an open-source gateway that explicitly markets the exact behaviors you need: mask-and-restore anonymization, policy-based routing, and automatic diverting of high-risk requests to a private/on-prem model. Its paper describes an open-source context-aware guardrails platform with data-leakage identification and redaction, and the GitHub project presents itself as an OpenAI-compatible AI Security Gateway for every LLM call. It is promising for your use case because it combines masking and private rerouting in one product. My only caution is that it is a comparatively new entrant, so I would validate it against more battle-tested components like Presidio and Envoy before making it the center of a production design. 

Memory, retrieval, and context minimization
LangGraph gives you a very practical way to keep memory scoped. Its docs distinguish thread-scoped short-term memory from namespace-scoped long-term memory, with persistent storage in JSON documents and support for database-backed stores such as PostgreSQL. That makes it well suited for a policy such as “tenant X memory can be used locally, but not sent to external providers unless it has been sanitized and risk-labeled.” 

Letta is useful when you want fine-grained control over what is actually in context. Its docs describe agents as a combination of system prompt, memory blocks, messages, and tools, and explain that memory blocks can be attached or detached from agents and pinned into the system prompt. That is a powerful abstraction for privacy because you can treat public, private, and shared memory as different attachable blocks instead of dumping everything into one conversation transcript. 

Mem0 is a credible open-source memory service if you want a drop-in server rather than a framework-level pattern. Its repository is Apache-2.0 licensed, supports self-hosting with auth enabled by default, and explicitly differentiates local library use, self-hosted server deployment, and cloud platform use. The project also supports user, session, and agent state, which maps well to privacy labels and retention policies. 

MemMachine is worth watching if you want newer memory research implemented as OSS. Both the paper and repository frame it as an open-source memory layer that stores raw episodes, reduces lossy LLM-based extraction, and emphasizes contextualized retrieval. For privacy-sensitive routing, its “ground-truth-preserving” design is attractive because it keeps original conversations local and avoids unnecessary re-summarization. 

For local retrieval, I would strongly prefer Qdrant over pushing vectors or retrieved chunks into a SaaS layer. Qdrant supports local mode in the Python client, persistent on-disk storage, payload filtering, and guidance on strict mode for filtering behavior. If you pair Qdrant with LlamaIndex, you gain metadata filters and node postprocessors that can remove or replace node content before response synthesis. That is exactly the kind of “least-context necessary” retrieval discipline that reduces leakage risk in RAG-backed routing flows. 

Hard egress and policy enforcement
Because vLLM Semantic Router already uses Envoy and ExtProc, the most natural enforcement point is an Envoy external processing service that can read and modify request bodies before they reach a SaaS backend. Envoy’s official docs are explicit that ext_proc can access and modify request headers, bodies, and response bodies through a gRPC external processor. That means you can place pseudonymization, placeholder restoration, and output scanning directly in the L7 request path. 

For allow/deny policy, OPA-Envoy is a strong fit. OPA’s docs describe the Envoy plugin as an external authorization service that evaluates request context, including HTTP request content, and returns fine-grained policy decisions. In your case, that means a redaction service can annotate a request with risk labels, and OPA can deterministically decide whether the request is allowed to leave the cluster, must be sanitized first, or must stay local. 

You should also make bypassing the privacy path impossible at the network layer. Kubernetes documents a default-deny egress policy; Cilium provides an egress gateway so selected pods’ outbound traffic is forced through designated gateway nodes; and Istio provides service-mesh policy and traffic control for inter-service communication. For this problem, the operational rule should be: only the sanitizer/gateway pod is allowed outbound internet access to OpenAI, Anthropic, or any other SaaS provider. Everything else is default-deny. 

Reference architecture for vLLM Semantic Router
The cleanest open-source design is a local-first, two-lane router.

The local lane handles anything that is high-risk, secret-bearing, tenant-confidential, or attached to private memory. vLLM Semantic Router already has local signal extraction, PII token labeling, and safety classification surfaces, so use those signals to keep risky traffic on a local vLLM, MLX-LM, or other on-prem model. This is also where local memory and RAG should live: LangGraph or Letta for scoped memory, Qdrant for filtered retrieval, and LlamaIndex-style postprocessors to trim context before inference. That architecture aligns with the router’s privacy-oriented design goals and with recent memory-augmented routing research showing that better knowledge access often matters more than escalating to a larger remote model. 

The sanitized SaaS lane is for prompts that are permitted to leave the cluster only after local transformation. In that path, the request should pass through a local Presidio-based service, ideally augmented with GLiNER or GLiNER2-PII for higher recall and custom entities. The sanitizer should replace detected values with deterministic placeholders or pseudonyms, store the mapping locally, and forward only the sanitized prompt to the remote provider. On the response path, output scanners such as LLM Guard, NeMo Guardrails, or a second Presidio pass can confirm that the model did not reconstruct unsafe content before placeholder restoration happens inside the cluster. OpenGuardrails and LiteLLM both support versions of this gateway pattern, but the privacy-critical intelligence should stay in the local sanitizer and policy engine. 

The third piece is egress enforcement. Envoy ext_proc can do the body rewrite; OPA can decide whether the request is allowed; and Kubernetes/Cilium/Istio can enforce that all outbound SaaS traffic traverses that gateway. Without that hard egress layer, any application component, notebook, or debug utility can silently bypass your redaction flow and invalidate the entire privacy story. 

A final design rule is that logs, traces, caches, and semantic memory must be sanitized too. vLLM Semantic Router explicitly includes semantic caching and memory surfaces, and CoT privacy work shows that traces and analytics stores can become persistent sensitive datasets. LiteLLM’s logging_only mode exists for exactly this kind of split, and Presidio even documents telemetry redaction as a deployment pattern. In other words, do not protect only the model call; protect the whole workflow state. 

Practical recommendations and limitations
If I were implementing your examples now, I would choose this stack first: vLLM Semantic Router for local classification and routing, Presidio + GLiNER/GLiNER2-PII for pre-egress redaction, Envoy ext_proc or LiteLLM as the OpenAI-compatible gateway, OPA for allow/deny policy, Cilium or Kubernetes default-deny egress for network enforcement, and LangGraph or Letta plus Qdrant for local memory and retrieval. That combination is fully open source, fits the Athena architecture well, and cleanly demonstrates that semantic routing can be privacy-preserving without sending raw prompts to OpenAI or Claude just to decide whether they should be used. 

If you want a higher-level gateway specifically for masking plus private rerouting, OpenGuardrails is the most directly aligned OSS project I found. If you want a more established and transparent route, build the gateway yourself around Envoy + Presidio + OPA, and treat NeMo Guardrails, Guardrails AI, and LLM Guard as secondary validation layers rather than primary DLP engines. 

The biggest limitation in the current open-source landscape is that there still is not a single, battle-tested OSS project that natively combines local sensitivity scoring, reversible redaction, memory governance, output validation, and network-enforced SaaS routing in one integrated vLLM-SR-specific package. The good news is that the components already exist and are mature enough to compose. The less good news is that you will still need to define your own policies for false negatives, placeholder formats, restoration safety, tenant boundaries, cache retention, and what counts as “too sensitive to ever leave the cluster.” Research like PPRoute, PRISM, Anonymous-by-Construction, and the CoT privacy papers strongly supports that direction, but the final production quality still depends on how well those OSS pieces are integrated and tested in your environment. 

Open questions and limitations
The main unresolved technical questions are not whether open-source tools exist—they do—but how you want to define policy at the edges:

whether low-risk prompts should be sent externally in sanitized form, or whether only fully non-sensitive prompts may leave the cluster at all;
how reversible your pseudonymization should be for code, identifiers, and internal product names;
whether your semantic cache and memory layer should store raw data, salted placeholders, or only derived embeddings; and
how you will benchmark false negatives for your own entity taxonomy, because Presidio itself explicitly warns that automated detection is not sufficient by itself.
